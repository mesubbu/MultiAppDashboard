# UI Gaps & Recommendations

Findings below are based on the combined screen inventory, journey map, and audit matrix. The emphasis is on journey completeness, operational safety, discoverability, and platform fit.

| Finding ID | Category | Affected Screen ID(s) | Description of the problem | Recommended action |
|---|---|---|---|---|
| GAP-001 | Missing | SCR-001 | The authentication flow has sign-in and MFA, but no explicit password reset, account recovery, or MFA recovery screen. This creates a dead end for legitimate operators who cannot complete login. | Add an auth recovery branch with password reset/request help, backup MFA handling, and clear operator support escalation. |
| GAP-002 | Missing | SCR-001, SCR-003, SCR-004, SCR-013 | The default post-login path appears to land on Overview, but Overview is not the best home for every role. Tenant-scoped and read-only users need a role-appropriate landing experience. | Introduce role-based default landing and a first-session “choose your workspace” rule tied to permissions and last-used scope. |
| GAP-003 | Missing | SCR-003, SCR-013, SCR-015, SCR-016 | There is no dedicated incident-detail screen that correlates overview alerts, live events, observability signals, and audit evidence into one narrative. Users must manually stitch evidence together. | Add an Incident Detail workspace that aggregates related event, service, model, and audit context with a stable incident timeline. |
| GAP-004 | Missing | SCR-019 | Research Operations needs a dedicated run-detail/results drill-down for collected documents, embeddings, source metadata, and degraded-run diagnostics. Without it, research review is shallow. | Add a Research Run Detail screen or drawer with artifact lists, failure diagnostics, and lineage into insights/recommendations. |
| GAP-005 | Missing | SCR-021, SCR-022 | Recommendations culminate in outcome capture, but there is no clearly defined recommendation approval/drill-down surface before action or feedback. That weakens operator trust for high-impact suggestions. | Add a Recommendation Detail screen with rationale, affected entities, supporting signals, recommended next step, and approval/feedback controls. |
| GAP-006 | Missing | SCR-017 | System Settings is governance-sensitive, yet the current surface appears review-only and lacks staged edits, diff review, rollback intent, or approval affordances. | Expand Settings into a managed change flow: draft edits, compare before/after, require confirmation, and route outcomes into Audit. |
| GAP-007 | Missing | SCR-003 to SCR-017 | Role-gated modules rely heavily on hidden navigation and API-level denial, but there is no dedicated “access denied / request elevated access” screen. This creates ambiguous failure states. | Add a reusable forbidden-state screen with the missing permission, safe next destinations, and access-request guidance. |
| GAP-008 | Missing | SCR-003 | The topbar exposes a live alert count, but there is no alert inbox or notification center screen. Users can see that alerts exist without a clear place to manage them. | Add an Alerts Inbox drawer/page with severity, ownership, status, and deep links into Overview, Events, Observability, and Audit. |
| GAP-009 | Redundant | SCR-003, SCR-014, SCR-015 | Overview, Analytics, and Observability all present overlapping KPI/health information. Without sharper boundaries, users may not know which screen is canonical for a given question. | Define an information hierarchy: Overview for triage, Analytics for trends, Observability for incident evidence. Reuse components but reduce duplicate content blocks. |
| GAP-010 | Redundant | SCR-013, SCR-016 | Events and Audit are both time-based operational histories with separate filter mental models. This increases cognitive load and weakens cross-screen consistency. | Standardize filter language, timeline structure, and export behavior; consider a shared timeline foundation with “operational” vs “governance” modes. |
| GAP-011 | Complexity | SCR-007 | The Agents console combines board, detail view, logs, task history, decisions, control actions, and recommendation snippets in one dense workspace. It is likely to overwhelm novice and intermediate operators. | Split Agents into progressive layers: fleet overview, agent detail, and advanced operations tabs/drawers. Preserve deep links into a selected agent state. |
| GAP-012 | Complexity | SCR-008, SCR-019, SCR-020, SCR-021 | The orchestration-to-research-to-recommendation lifecycle spans multiple datasets, roles, and outcomes, but the current IA does not clearly stage those steps. This raises implementation and usability risk. | Design this capability as a connected workbench with list → detail → result → feedback transitions instead of isolated high-density pages. |
| GAP-013 | UX | SCR-007, SCR-010, SCR-017, SCR-022, SCR-025 | High-impact mutations rely on inline actions and toast feedback, but many of them should present impact review, confirmation, and next-step guidance. Toasts alone are too ephemeral. | Add confirmation modals/sheets for risky actions, include impact summaries, and persist a post-action status panel or audit reference. |
| GAP-014 | UX | SCR-004, SCR-005, SCR-006, SCR-016 | Table-heavy admin screens mix filtering and editing states without a strong saved/unsaved model. Users may not know what changed, what failed, or what still needs confirmation. | Add explicit dirty-state indicators, save state messaging, inline validation, and optional batch-save patterns where multi-row edits are likely. |
| GAP-015 | UX | SCR-023, SCR-024, SCR-001 | The recovery story is incomplete for session expiry vs transient fetch failure vs hard service outage. A generic retry screen is not enough for trust-critical admin workflows. | Differentiate recovery states by failure type and provide context-specific next actions: retry, re-authenticate, open status page, or view cached last-known state. |
| GAP-016 | Accessibility | SCR-012, SCR-013, SCR-015, SCR-025 | Graph exploration, live event streaming, auto-refreshing charts, and toast-based feedback can be difficult for keyboard, screen-reader, and motion-sensitive users. | Provide equivalent tabular/list views, stronger keyboard paths, pause controls by default for live regions, reduced-motion behavior, and persistent status summaries. |
| GAP-017 | Accessibility | SCR-004, SCR-005, SCR-006, SCR-007 | Dense inline-edit controls inside data tables risk poor focus order, ambiguous labels, and weak error association. Complex operator surfaces need stronger form semantics than standard tables provide. | Use explicit row edit modes, linked labels/help text, per-field error messaging, and deterministic focus return after save/cancel. |
| GAP-018 | Mobile | SCR-002, SCR-018, SCR-007, SCR-012 | Context switching, assistant usage, agent operations, and graph exploration are desktop-shaped experiences today. On smaller viewports, critical surfaces may be hidden or become unusable. | Add a mobile IA: nav drawer, scope sheet, bottom-sheet assistant, simplified agent cards, and a non-canvas graph fallback. |
| GAP-019 | Mobile | SCR-013, SCR-015, SCR-016, SCR-017 | Event, observability, audit, and settings views are table-heavy and detail-dense. A straight responsive shrink will create scan and tap-target problems. | Introduce stacked cards, drill-in drawers, pinned key fields, and mobile-first export/change workflows rather than full-width desktop tables. |
| GAP-020 | Missing | SCR-008, SCR-019, SCR-020, SCR-021 | Several spec-critical capabilities are required by the platform but are not clearly represented in the current dashboard navigation structure. This creates discoverability and delivery risk. | Promote these capabilities into explicit routes/navigation groups with consistent labels, ownership, and cross-links from Agents and Assistant flows. |
